## 0. 基本

技術書の編集者として、以下の点については確実に確認すること。

- 技術的な誤りがないこと。
- 正しい日本語であること。誤字脱字がないこと。
- 表記ゆれがないこと。
- 各ファイルにおける局所的なレビューを確実に行いつつ、書籍全体でみた統一性・バランス・流れの良さなどについても注意して確認すること。

## 1. コンセプトとターゲット（Tone & Target）
- **前提知識の維持**: 読者は「Python の基礎（リスト、辞書、クラス）は知っているが、言語処理系は初めて」という想定。説明にギャップがないか。難易度が急激に跳ね上がっていないか。
- **トーン＆マナー**: 堅苦しい学術的な表現は避け、ユーモアを交えた「伴走型・対話型」の親しみやすいトーンを維持できているか確認する。
- **標準的な用語**: 読者が発展的な学習を行うときに役立つよう、標準的な用語を使用する。
- **比喩の活用**: 難しい概念が日常的な比喩（Excel と CSV など）に落とし込まれているかを評価する。
- **豆知識・うんちく**: 適切なタイミングで適切な豆知識やうんちくを入れる。入れておくべき豆知識やうんちくがあったら提案してください。

## 2. 構成とペース配分（Pacing & Motivation）
- **インクリメンタルな開発**: 一気に複雑な実装をせず、少しずつ機能を追加して「すぐ動く」達成感を担保できているか。
- **無理のない進行速度**: 一度に多すぎる概念を導入しようとしていないか。気に複雑な実装（例: 環境のネストとクロージャの同時実装など）が行われていないか。難易度のギャップが大きいと判定された場合、中間にどのような小さな実装ステップ（あるいは比喩や豆知識のコラム）を挟むべきか、具体的な構成案を提示してください。
- **コードの再現性**: 読者が迷わず写経できるよう、追加するコードが `diff` 形式等で明確に示されているか。

## 3. タイポグラフィと書式ルール（Typography）
Markdown の記述において、以下の使い分けが徹底されているか厳密にチェックする。
- **インラインコード（\`code\`）**: ソースコード上のキーワード、型、変数名、AST のタグ（例: \`eval()\`、\`bool\`、変数 \`n\`、\`add\`、\`if\`）。識別子を表しているときはダブルクォーテーションで囲まず単に \`add\` と書く。文字列を表している場合はクォーテーションも含める。
- **数式（$math$）**: 数学・アルゴリズム上の抽象的な変数（例: `$n$` の階乗）。
- **通常テキスト**: 構文の概念そのもの（例: if 文、順次実行）。単独の数字（例: 1、100）。
- **見出しのスペース**: 章タイトルは `# 第 1 章　最小の評価器` のように、全角スペースをセパレータとして用いる（textlint 回避のため）。

## 4. 技術的な正確性と割り切り（Technical Scope）
- **やらないことの明確化**: エラー処理、堅牢性、性能、メモリ管理などを「実装しない」という割り切りが読者に伝わり、混乱を防げているか。
