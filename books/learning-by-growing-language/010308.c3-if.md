今度は `if` です。

まだ仮想マシンには条件によって動作が変わる命令がなくて、一直線に実行が進むだけなので `if` のような動きはできません。そこで `jump_if_false` という命令を導入します。これは、「スタックから値をひとつ取り、偽だったら〇〇番地に飛ぶ」という命令です。

番地というのは命令の置かれた位置を表すものです[^memory-address]。下記の中間コードで言うと、 0 から 5 の数をアドレスと呼び、`add` は 3 番地にある、とか `halt` のアドレスは 5 番地、などと言ったりします。Toil の場合は `_code` で示される中間コードの配列のインデックスと一致します。

[^memory-address]: 普通に使うときはもう少し意味が広くて、メモリ上のデータの位置を表します。

```py
 0: ('enter_scope',)
 1: ('get', 'a')
 2: ('const', 3)
 3: ('add',)
 4: ('leave_scope',)
 5: ('halt',)
```

 `jump_if_false` を使うと「`2 == 2` ならば `3` を `print` する」という処理は以下のように書けます。

```py
 0: ('const', 2)
 1: ('const', 2)
 2: ('equal',)
 3: ('jump_if_false', 6)
 4: ('const', 3)
 5: ('print',)
 6: ('halt',)
```

`('jump_if_false', 6)` では、`2 == 2` が偽だったら 6 番地、つまり `('const', 3)` と `('print',)` をすっ飛ばして `('halt',)` にジャンプします。実際には真なのでそのまま続きの命令に進み、`3` を `print` します。「`2 == 2` ならば `3` を `print` する、そうでなければ `print` はスキップする」という感じですね。 `jump_if` ではなく  `jump_if_false` を作った理由を感じていただけたでしょうか？

しかし `jump_if_false` だけでは命令が足りません。やっぱり  `jump_if` が必要だろう？ハズレです。 ただの `jump` が必要です。というのは Toil の `if` には `else` があるからです。

「`2 == 2` ならば `3` を `print` する、そうでなければ `4` を `print` する」の中間コードは以下のようになります。`3` を `print` したあと、`4` を `print` するコードをスキップするために `jump` を使っています。

```py
 0: ('const', 2)
 1: ('const', 2)
 2: ('equal',)
 3: ('jump_if_false', 7)
 4: ('const', 3)
 5: ('print',)
 6: ('jump', 9)
 7: ('const', 4)
 8: ('print',)
 9: ('halt',)
```

`jump` のような動作をどう実現するのかと言うと、`_ip` を変更するだけです。`jump_if_false` はスタックの値が真でなければ、という条件がつきます。

```diff py 
     def execute(self):
         while (inst := self._code[self._ip]) != ("halt",):
             self._ip += 1
             match inst:
                 ...
                 case ("get", name): self._stack.append(self._env.val(name))
+                case ("jump", addr): self._ip = addr
+                case ("jump_if_false", addr):
+                    if not self._stack.pop(): self._ip = addr
                 case ("print",):
                     print(self._stack.pop()); self._stack.append(None)
                 ...
             ...
```

`_expression()` はそろそろ説明不要でしょう。

`_current_addr()` はこれから追加しようとしている命令の番地を返します。命令は配列の最後に `append()` されるので、配列の長さが番地となります。`_set_operand(self, ip, operand)` は `ip` 番地の命令のオペランドを `operand` に書き換えます。

では `_if` 本体を見ていきましょう。ここからが本節のハイライトです。「`2 == 2` ならば `3` を `print` する、そうでなければ `4` を `print` する」つまり `if 2 == 2 then print(3) else print(4) end` のコンパイルを例にとって説明します。

```py
    def _if(self, cond_expr, then_expr, else_expr):
        self._expression(cond_expr)
```

まず Cond 式をコンパイルします。実行すると Cond 式の結果がスタックに残っているはず。さきほど例示した中間コードで言えばこの部分です。

```py
 0: ('const', 2)
 1: ('const', 2)
 2: ('equal',)
```

```py
        else_jump = self._current_addr()
        self._emit("jump_if_false", None)
```

これは何をやっているのでしょうか？出力したいのはこういうコードです。スタックに残っている Cond 式の結果が偽ならば 7 番地へ。

```py
 3: ('jump_if_false', 7)
```

ところが、現時点では何番地に飛べばよいのかわかりません。そのため、飛び先はとりあえず `None` としておきます。実際にはこんな形で保存されているというわけですね。

```py
 3: ('jump_if_false', None)
```

あとで `_set_operand()` を使って書き換えられるように、`else_jump` に現在の番地を覚えておきます。2 番地まで命令を保存しているので `self._current_addr()` は 3 で、`else_jump` も 3 になります。

```py
        self._expression(then_expr)
```

Then 式をコンパイルします。中間コードはこうです。

```py
 4: ('const', 3)
 5: ('print',)
```

```py
        end_jump = self._current_addr()
        self._emit("jump", None)
```

こういう中間コードを出力したい。以下同文です。

```py
 6: ('jump', 9)
```

ここでオペランドの書き換えです。これから Else 式のコンパイル結果を出力するので、`jump_if_false` ではここに飛びたい。

```py
        self._set_operand(else_jump, self._current_addr())
```

`else_jump` は 3 なので、3 番地の命令のオペランドを書き換えます。6 番地まで `_code` に命令がたまっているので `self._current_addr()` は 7 です。つまり、以下のようにいったん作った 3 番地の命令が、

```py
 3: ('jump_if_false', None)
```

このように書き換えられます。狙い通り！

```py
 3: ('jump_if_false', 7)
```

このように、番地が確定してから戻ってアドレスを書き換える技をバックパッチと言います。今日はこれだけ覚えましょう。

```py
 7: ('const', 4)
 8: ('print',)
 9: ('halt',)
```

あとはわかりますか？

```py
        self._expression(else_expr)
        self._set_operand(end_jump, self._current_addr())
```

Else 式をコンパイルすると、`jump` の飛び先が確定するのでさっきと同様にバックパッチします。以上です。

## コラム：バックパッチ以外の方法とコンパイルのパス

ソースコードや AST などを最初から最後まで操作することを「パス」と言います。

Toil のコンパイルでは、バックパッチを使って AST を 1 周するだけ、つまり 1 パスでコンパイルを終わらせることができますが、コンパイルが終わった時点でジャンプの飛び先が固定されてしまうため、その後の最適化が難しいという弱点があり、小規模なコンパイラで用いられます。

バックパッチは未定の飛び先アドレスを埋めるための代表的な手法ですが、唯一の手法ではありません。ほかにも以下のような手法があります。

* ソースコードを 2 周して、1 周目は番地を調べるだけにし、2 周目でその番地を使って命令を組み立てる
* 飛び先を番地ではなくラベルで表しておき、あとでラベルを番地に置き換える
* アセンブリ言語にラベルで書いてあとはアセンブラに任せる

どれもプログラム全体を 2 周見て回る必要があります。現代ではコンピュータのメモリ容量や速度が飛躍的に向上したため、本格的なコンパイラでは AST や中間コード全体をメモリに保持したまま何度も走査して、飛び先を求めるだけでなく最適化なども行うのが一般的です。

パスを数えるのをコンパイル時以外にも広げると、中間コード式の Toil では３つのパスがあります。それぞれのパスは入力をまとめて全部受け取り、すべて処理してから出力をまとめて次のパスに渡しています。

* ソースコードを読んでトークン列にする
* トークン列を読んで AST にする
* AST を読んで中間コードにする

昔のコンピュータではこれらの入力・出力データを全部まとめてメモリに置いておくことが困難で、ソースコードを少しずつ読んではその場で最終的なコードを少しずつ出力していくものもありました。これが 1 パスコンパイラです。パスの出力結果をディスクに保存することもできましたが、昔はディスクの速度も（現在と比べると驚くほど）遅かったため、コンパイルが遅くなってしまいます。その点でも 1 パスコンパイラに優位性がありました。

ただし、1 パスでコンパイルを終わらせるためには言語自体に制約を加える必要がありました。C 言語では関数定義より先に呼び出しのコードを書く場合プロトタイプ宣言が必須となりますが、これなどはその名残です。
