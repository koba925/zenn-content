変数の定義と代入を扱えるようにします。

`<変数名> := <値>` で変数を定義することとし、`("define", [<変数名>, <値>])` という AST に変換、`<変数名> = <値>` で変数に値を代入することとし、`("assign", [<変数名>, <値>])` という AST に変換します。

`define` や `assign` は `<値>` を返しますので、`<変数名> := <値>` や `<変数名> = <値>` も値を返すことになります。これにより `a := b := 2` のように連鎖した書き方も可能になります。このとき大事なのは `a := b := 2` は `a := (b := 2)` であって `(a := b) := 2` ではない、つまり右結合だということです。

そのため `_binary_left()` を使うことはできず、右結合用の処理を新たに書く必要があります。ここがこの節のキモです。

優先度は比較よりもさらに下とします。`a := 2 == 3` と書いた時には `a := (2 == 3)` という意味になってほしいことが多いだろう、ということです。`(a := 2) == 3` のように解釈してほしいこともそれほど特殊なケースではありませんし[^actually]、カッコをつければどちらの意味にもできるのでここは言語設計者で決めることができるところですが、こういう目につきづらいところの仕様が他の言語と異なっていると利用者のつまづきの元にもなるのでそういう点も考慮に入れる必要があります。Toil ではあんまり重視していませんが！

[^actually]: もともと `type()` で囲まれているので余分にカッコをつける必要はありませんが、`type(op := self._current_token()) is str` はそういう気持ちで書いています。

といったあたりを考慮していつものアレを書くとこうなります。

・・・
「式」は「定義・代入」、
「定義・代入」は「比較」のあとに（`:=` または `=` ＋「定義・代入」）が続いたもの
「比較」は「足し算・引き算」のあとに・・・
・・・

「定義・代入」の中に「定義・代入」が登場しているところが注意点です。再帰的な決まりなのでコードも再帰になります。

「比較」のあとに、というところでアレ？と思ったでしょうか？`a := 2` のような形なんだから最初に来るのは `a` のような識別子のはずで、「比較」じゃないのでは？と。おっしゃるとおりでございます。第 2 部以降で左辺が単なる識別子ではないケースが出てくるのでこのような形にしています[^impure-motivation][^impure-motivation2]。左辺に `2 < 3` みたいなものを書いてしまっても最終的にはエラーになるので大きな実害はありません。

[^impure-motivation]: 後述する `_binary_right()` というメソッドを汎用的に書きたいというのもありますがこれは不純な動機かもしれません。`:=` や `=` は左辺が普通に評価される「式」でないという時点で普通の演算子ではないので。ぶっちゃけ `_binary_left()` があるんだから `_binary_right()` も作りたい、くらいの話です。
[^impure-motivation2]: さらに言うと、汎用と言いつつ Toil では `_define_assign()` 以外に `_binary_right()` を呼ぶことはありません（今後の拡張でも）。でも累乗（Python の `**`）を実装したいと思ったときには使えるのでお使いください（累乗は右結合が普通）。

ハナから `_binary_right()` にくくりだしていますが、初めてだとわかりづらいと思うので元の形に復元？して考えましょう。

```py
    def _define_assign(self):
        ops = {":=": "define", "=": "assign"}

        left = self._comparison()
        if type(op := self._current_token()) is str and op in ops:
            self._current_and_advance()
            return (ops[op], [left, self._define_assign()])
        else:

```

まず `_comparison()` で「比較」（実際は識別子のはず）を読み、次のトークンが `":="` か `"="` だったら、ひとつ進めて `_define_assign()` で「定義・代入」を読み、`define` や `assign` の AST を組み立てて返します。`":="` でも `"="` でもなかったらさっき読んだ「比較」をそのまま返します。

すんなり読めたでしょうか？`_define_assign()` の中から `_define_assign()` を呼んでいるところ、つまり再帰的に自分自身を呼び出しているところでちょっと戸惑う人がいるかもしれませんが、実はそれほど難しく考える必要はありません。

`a := b := 2` で考えてみます。`_define_assign` が呼ばれると、まず `_comparison()` で `a` が読まれ、`left` に記録されます。次のトークンは `":="` なので `if` の中に進み、トークンをひとつ進めます。この時点で、これから `b := 2` を読むという状態になっています。ここで `_define_assign()` を呼んでいますが、自分自身を・・・と考え込まずに `_define_assign()` は「定義・代入」を読んでその AST、つまり `("define", ["b", 2])` を返してくれるはずだ、と信じます。すると、`(ops[op], [left, self._define_assign()])` は `("define", ["a", ("define", ["b", 2])])` になります。ちゃんと `a := b := 2` を表す AST になっています。

なんか足元が定まってないような気がしたら、もう 2 回同じように考えると今度は安心できるはずです。どうでしょうか？まだちょっと・・・と思ったら「コラム：再帰の読み方」へ。

実行例に進みます。

```py

    print(toil.walk(r""" a := 2 """))  # -> 2
    print(toil.walk(r""" a """))  # -> 2
    print(toil.walk(r""" a = 3 """))  # -> 3
    print(toil.walk(r""" a """))  # -> 3


    print(toil.ast(r""" b := c := 4 """))  # -> ('define', ['b', ('define', ['c', 4])])
    print(toil.walk(r""" b := c := 4 """))  # -> 4
    print(toil.walk(r""" b """))  # -> 4
    print(toil.walk(r""" c """))  # -> 4

    print(toil.ast(r""" b = c = 5 """))  # -> ('assign', ['b', ('assign', ['c', 5])])
    print(toil.walk(r""" b = c = 5 """))  # -> 5
    print(toil.walk(r""" b """))  # -> 5
    print(toil.walk(r""" c """))  # -> 5

    print(toil.ast(r""" a := 2 == 2 """))  # -> ('define', ['a', ('equal', [2, 2])])
    print(toil.walk(r""" a := 2 == 2 """))  # -> True
    print(toil.walk(r""" a """))  # -> True

    # toil.walk(r""" not_defined = 3 """)  # -> Undefined variable
```

## コラム：再帰の読み方

再帰は難しい、わからない、と書いてあるのをよく見かけます。そういうひとにヒントをふたつ伝えたい。

まず、再帰のあるコードを読むのは（わかるまでは）簡単ではないということです。ラノベを読むようにさらさら読もうとしてわからなくても普通です。1 行ずつゆっくり読みましょう。

ここからがふたつめのヒントで本題です。

**おすすめな読み方**

1 行 1 行読んでるんだけどわかった気がしない、というのは、再帰を全部たどろうとしているからかもしれません[^reality]。もしそうだったら、再帰する呼び出しは正しいと信じて、それ以上掘り下げない、という読み方を試してみてください。

たとえば Python で `sum([1, 2, 3])` っていうコードがあったら `sum` の中身はなんだろう、などと気にしないで「ちゃんと足してくれるだろう」と思って読みますよね（たいていは）。それと同じように読んでみる、ということです。

もう一つ例を。"2.7 カッコ" で以下のコードが出てきたときはどう感じたでしょうか？「うわ再帰だ難しそう」「これ一体どうなるんだ？」と思ったでしょうか。

```py
    def _group(self):
        self._current_and_advance()
        expr = self._expression()
        self._consume(")")
        return expr
```

あ、カッコの中は「式」だから `_expression()` 呼ぶんだな、 `_expression()` が式を読んでくれるんだな、くらいですんなり受け入れられたのではないでしょうか[^reality2]？実際にはいくつも他のメソッドを挟んだ、より複雑な再帰になっているわけですが。

これはひとえに `_expression()` が目につかないことによるのではないかと思います。`_define_assign()`（復元したほう）では、呼び出すところと呼び出されるところがすぐ近くにあるのでつい気になってしまうだけで、半分くらいは錯覚なのではないかと。なので、`:=` の右側は「定義・代入」だから `_define_assign()` を呼ぶんだな、`_define_assign()` が「定義・代入」を読んでくれるんだな、と思えばいいんですよ、というのがここで言いたいことです。本文の

> `_define_assign` は「定義・代入」を読んでその AST、つまり `("define", ["b", 2])` を返してくれるはずだ、と信じます。

と書いているのがそれにあたります。

[^reality]: 私の頭の中にいる再帰が苦手な人のイメージです。「再帰は難しい、わからない」というのも、教える側のひとが書いてるのしか見たことがありません。もしかするとこのコラムの対象読者はこの世に存在しないかもしれません。
[^reality2]: これも想像上のイメージです。いやそうじゃないんだよ、という方がいらしたらぜひ教えてください。

**やってみる**

慣れるためにずっと簡単な例からやってみましょう。0 から n までの整数の和を返す以下の関数で説明します。

```py
def to_n(n):
    if n == 0: return 0
    return n + to_n(n - 1)
```

`to_n(5)` の呼び出しを考えます。`5 == 0` ではないので `return` で `5 + to_n(4)` を返そうとします。`to_n(4)` は正しく 10 を返してくれる、と信じると、`to_n(5)` は 15 になります。正しいですね。かんたん！以上です！

それでいいの？ずーっとぐるぐるしない？という心配は無用です。呼び出しを続けていくと最後は `n` が `0` になって `return 0` が `0` を返すところまで行きつきます。これは正しい。あとは全部遡って芋づる式に正しいというわけです。再帰が止まるところを「基底」と言います。基底を確認することも大事ですが、誰かの書いたコードを読むときには基底もちゃんと定義されてるはずです。そうでないといつまでたっても再帰が止まらないので。

この考え方は再帰を書くときにも応用できます。`to_n(5)` を求めたいとき、何もないところから考えるのではなく、`to_n(4)` がわかっているものとして考えればいいわけです。もっと一般化していえば、`to_n(n)` を求めたいときに `to_n(n-1)` はわかっているものとして考えてよいということです。学校で習った数学的帰納法を思い出した方もいるかもしれません。同じしくみです。あと、自分で書くときは基底を忘れずに定義しましょう。大事です。

この読み方に慣れると再帰を読むのが（書くのも）うんと楽になります。むしろ再帰で書いてある方が素直、と思えるくらいに。ちょっとややこしいことを書こうとしてこれは再帰で書けるやつだな、と思って「n - 1 のときはわかってるものとして～～」と書いていくと思ったより単純で、ほんとにこれで動くのか？と思って半信半疑なのに動かすと動いてしまう、ということもしばしばです。普通のプログラミングでは動くと思ってても動かないのに！何それ！

**始めからたどっていく方法**

でもやっぱり心配だ、という場合は浅い再帰から全部試していってみましょう。腹落ちするまでやっておくのもよいかと思います。いちばん浅いのは基底で、`to_n(0)` つまり 0 ですね。`to_n(1)` は `1 + to_n(0)` ですから、`to_n(0)` の結果が使えて 1 + 0 = 1 となります。続きも芋づるで以下のようになります。

```
to_n(0) = 0
to_n(1) = 1 + to_n(0) = 1 + 0 = 1
to_n(2) = 2 + to_n(1) = 2 + 1 = 3
to_n(3) = 3 + to_n(2) = 3 + 3 = 6
to_n(4) = 4 + to_n(3) = 4 + 6 = 10
to_n(5) = 5 + to_n(4) = 5 + 10 = 15
```

実際の呼び出しではこれを下の方からたどっているわけです。ひとつ前の結果がわかっていればちゃんと計算できる感じがつかめたでしょうか？

**全部遡ってみる**

やっぱり「再帰を全部たどる」をそのままたどってみたい？それもありだと思います。両方わかってるのが理想的でしょう。この場合、再帰を追いかけるとき関数全体を見ていると頭の容量が足りなくなってしまうので、特に再帰呼び出しの部分に注目し、呼び出しのときどんな呼び出しをしているかを追いかけます。`return` の部分ですね。さっきやったのをひっくり返してつなげていく感じです。

```
  to_n(5)
= 5 + to_n(4)
= 5 + (4 + to_n(3))
= 5 + (4 + (3 + to_n(2)))
= 5 + (4 + (3 + (2 + to_n(1))))
= 5 + (4 + (3 + (2 + (1 + to_n(0)))))
```

`to_n(0)` は 0 ですからこうなります。

```
= 5 + (4 + (3 + (2 + (1 + 0)))) 
```

再帰はここで止まります。ここでおわりにしても OK なのですが、関数呼び出しから帰っていくところまで追いかけてみましょう。

```
= 5 + (4 + (3 + (2 + 1)))
= 5 + (4 + (3 + 3))
= 5 + (4 + 6)
= 5 + 10
= 15
```

再帰にもいくつかのパターンがあって、シンプルなものならこうやって最後までたどることができます。そうでないものもあって、フィボナッチ数列なんかは２・３回たどったらもう書ききれなく（書く気がしなく）なってくるでしょう。そういう場合も浅い再帰から書いていく方式なら書けることもあります。

最初に書いた通り、（感覚がつかめるまでは）簡単なものではないのでわかるまでいろいろ練習してみるとよいかと思います。再帰を徹底的にやってみようと思ったら [Scheme手習い](https://amzn.to/3z6j4aE) がおすすめです。ただしクセがあって万人にお勧めしますとも言えない。私は大好きですが。

**`_define_assign()` でもう一度**

理解と練習を兼ねて、Toil の `_define_assign()` をいろいろな読み方で読んでみましょう。

```py
    def _define_assign(self):
        ops = {":=": "define", "=": "assign"}

        left = self._comparison()
        if type(op := self._current_token()) is str and op in ops:
            self._current_and_advance()
            return (ops[op], [left, self._define_assign()])
        else:
            return left
```

「再帰する呼び出しは正しいと信じて、それ以上掘り下げない」読み方は本文に書きましたが、基底ケースの確認はサボってました。

基底ケースは `type(op := self._current_token()) is str and op in ops` が成り立たないケースで、たとえば `[2, "$EOF"]` を `_define_assign()` するといきなり基底ケースに入ります。

まず `self._comparison()` で `2` を読んで `left` に入れ、`if` の条件が成り立たないので `return left` が実行されて `2` が返ります。OK ですね。`a := b := ... := 2` のように長くつながっていてもいつかはこの基底ケースに届きます。

ここから下は短く書けるように、たとえば `a := b := 2` をトークン列にしたものを `_define_assign()` することを `DA(a := b := 2)` と書くことにします[^parser-combinator]。

[^parser-combinator]: オブジェクト指向だと属性に情報を覚えさせるのでこういうことを考えるときにちょっと書きにくいところがあって、ここでは引数の形で表現してみました。関数型言語では引数に全部情報があるので考えやすくなります。実際に構文解析を関数型で書くと「パーサコンビネータ」と呼ばれるものになります（すごく大ざっぱに言えば）。

すると基底ケースは以下のように書けます。

```
DA(2) = 2
```

ここから始めて「始めからたどっていく方法」で読んでみましょう。

次は `a := 2` です。

最初に `_define_assign`（これを A とします）が呼ばれると、まず `_comparison()` で `a` が読まれます。次のトークンは `":="` なので `if` の中に進みます。まだトークンは `":="` を見ているのでひとつ進め、今度は `_define_assign()` を呼んで「定義・代入」を読みに行きます。残りは `2` だけなので基底ケースにあたり `2` が返ってきます。すると `(ops[op], [left, self._define_assign()])` は `("define", ["a", 2])` になります。

言葉で書くと長いですね[^longer-in-words]。図式的に書いてみましょう。

[^longer-in-words]:逆に見ると、文章にすると長たらしくなることをコードではコンパクトに圧縮して書いているわけです。これが「ラノベを読むようにさらさら」とは読めない理由のひとつです。

```
  DA(a := 2)
→ ("define", ["a", DA(2)])
→ ("define", ["a", 2])
```

次は `b := a := 2` です。再利用をわかりやすくするため `a` と `b` を入れ替えました。`DA(a := 2)` を上の結果を再利用して書き換えます。文章による説明は抜きで。

```
  DA(b := a := 2)
→ ("define", ["b", DA(a := 2)])
→ ("define", ["b", ("define", ["a", 2])])
```

想定通りの AST になりました。あとはいくら伸びても同じです。

この読み方・書き方がわかれば「全部遡ってみる」もわかるはず。

```
  DA(b := a := 2)
→ ("define", ["b", DA(a := 2)])
→ ("define", ["b", ("define", ["a", DA(2)])])
→ ("define", ["b", ("define", ["a", 2])])
```

`to_n()` とは違って後半簡単になっていくプロセスはありません。これでできあがりです。

**まとめ**

再帰的に呼び出しているからと言って遡って追いかけようとせず、正しい値を返してくれると信じて読んでみるといいよ、たいていはそれで充分、というお話でした。普通の関数呼び出しだとそういう風に読むことも多いと思いますが、再帰だとすぐ目の前に呼び出したい関数があるわけで、つい読もうとしてしまう、ということかもしれません。

きっとこれで再帰の動きを少し具体的にイメージできるようになっていると思います。

再帰は使いどころで使うととても強力な道具です。複雑なものを一気に組み立てるのではなく、「1 段階だけ簡単なもの」を材料として目的のものを作るようにすることで、劇的に簡単に書けることがあります。ぜひマスターしてください。


## コラム：左再帰と右再帰

`_binary_left()` と `_binary_right()` はとてもよく似ています。

```py
    def _binary_left(self, ops, sub_elem):
        left = sub_elem()
        while type(op := self._current_token()) is str and op in ops:
            self._current_and_advance()
            left = (ops[op], [left, sub_elem()])
        return left

    def _binary_right(self, ops, sub_elem):
        left = sub_elem()
        if type(op := self._current_token()) is str and op in ops:
            self._current_and_advance()
            return (ops[op], [left, self._binary_right(ops, sub_elem)])
        else:
            return left
```

違いは `while` と `if`、あと `sub_elem` の呼び出し方法だけです。なぜこのような違いが生まれるのでしょうか？

`_binary_left()` は左結合の二項演算子用で、このようなときに、

> 「かけ算」は「プライマリ式」のあとに（`*`「プライマリ式」）が 0 個以上続いたもの、

`_binary_right()` は右結合の二項演算子用で、このようなときに使うのでした。

> 「代入」は「比較」のあとに（`=`「代入」）が続いたもの

（読みやすいように「かけ算・割り算・剰余」や「定義・代入」ではなく「かけ算」「代入」のみにしました）

そもそもここでの表現方法が違うわけで、これをコードに落とせば違うコードになるのは当然ではあります。

ですが、実は「かけ算」をこのように表現することもできるのです。「0 個以上」ではないことに注意してください。「かけ算」の定義の中に「かけ算」が出てきていて、これも再帰的な定義です。

> 「かけ算」は「かけ算」のあとに（`*`「プライマリ式」）が続いたもの

たとえば `2 * 3 * 4` は `2 * 3` という「かけ算」のあとに `* 4` が続いた形と読むことができ、`(2 * 3) * 4` と左結合の式が作れることがわかります。

「かけ算」のように、演算子の左側が再帰になっているものを左再帰、「代入」のように演算子の右側が再帰になっているものを右再帰と呼びます。

計算機科学の世界ではだいたい繰り返しよりも再帰のほうが扱いやすいことになっていて、できれば再帰でやりたいのです。なのになぜ左再帰の形を「0 個以上」のような繰り返しの形で表現するのでしょうか？

左再帰の形で `_binary_left()` を書いてみましょう。

```py
    def _binary_left(self, ops, sub_elem):
        left = self._binary_left(ops, sub_elem)
        if type(op := self._current_token()) is str and op in ops:
            self._current_and_advance()
            return (ops[op], [left, sub_elem()])
        else:
            return left
```

これではダメなんでしょうか？

`2 * 3 * 4` でやってみましょう。`_binary_left()` を呼ぶと、まず `_binary_left()` を呼び・・・ダメですね。なにもせずただ自分を呼ぶだけなので無限ループになってしまいます（実際にはスタックがあふれてエラーで止まる）。

ここが再帰下降構文解析の弱点で、これをカバーするために左再帰を繰り返しに書き換えているのです。2.7 の「コラム：その他の構文解析手法」で紹介したボトムアップの構文解析なら左再帰もそのまま書けます。

なお、右再帰では再帰する前に下位の要素や演算子を読んでいるため、無限に再帰することはないのでそのまま再帰で書けています。
