---
title: "2.2 数を読む（構文解析）"
---

この節では、字句解析器が出力したトークン列を読み、AST に変換する「構文解析」（Syntax Analysis）を行います。構文解析を行うプログラムは「構文解析器」ですが、英語では通常 Parser（パーサ）と呼ばれます。構文解析自体も「パース」「パースする」（parse）と呼ぶことも多いです。

構文解析の方法にはいろいろなものがありそれぞれに長所・短所がありますが、Toil では「再帰下降型構文解析」（Recursive Descent Parsing）という手法を用います。再帰下降型構文解析は、採用できる文法に多少制限がありますが、文法をほぼそのままコードにできて書きやすいという特徴があります。文法の制限といっても十分にプログラミング言語らしい文法にすることは可能で、メジャーな言語処理系でも再帰下降型の構文解析を行っているものもあります。


AST に変換するといっても今のところは `[<数>, "$EOF"]` という形のトークン列を受け取って `<数>` という AST を返すだけなんですが、字句解析同様基本的な構造を導入してしまいます。

`Scanner` クラスと `Environment` クラスの間に以下の通り `Parser` クラスを追加してください。

```diff py
 class Scanner:
     ...

+class Parser:
+    def __init__(self, tokens):
+        self._tokens = tokens
+        self._pos = 0
+
+    def parse(self):
+        expr = self._expression()
+        assert self._current_token() == "$EOF", \
+            f"Extra token @ parse(): {self._current_token()}"
+        return expr
+
+    def _expression(self): return self._primary()
+
+    def _primary(self):
+        match self._current_token():
+            case int(): return self._current_and_advance()
+            case invalid:
+                assert False, f"Invalid token @ _primary(): {invalid}"
+
+    def _current_token(self): return self._tokens[self._pos]
+
+    def _current_and_advance(self):
+        self._pos += 1
+        return self._tokens[self._pos - 1]

 class Environment:
     ...
```

パーツごとに説明します。

```py
class Parser:
    def __init__(self, tokens):
        self._tokens = tokens
        self._pos = 0
```

`Parser` クラスです。`_tokens` は字句解析の出力結果であるトークン列を覚えておくもの、`_pos` はいまトークン列の何番目の要素を読んでいるか覚えておくものです。

つぎはユーティリティ関数です。

```py
    def _current_token(self): return self._tokens[self._pos]

    def _current_and_advance(self):
        self._pos += 1
        return self._tokens[self._pos - 1]
```

`_current_token()` は現在位置のトークンを返します。

`_current_and_advance()` は現在位置のトークンを返すついでに現在位置をひとつ進めます。といっても `return` してから現在位置を進めることはできないので、ひとつ進めておいてからひとつ前の要素を返すという書き方になっています。現在位置のトークンの情報が不要でただ現在位置を進めたいときも使っています。ちょっとだけもったいないですけど。

ここからがメインの処理です。

ここでは、Toil のプログラムが以下のような形になっていると見ています。つまり Toil はそういう文法だということです。

「プログラム」は「式」、
「式」は「プライマリ式」、
「プライマリ式」は「数」

本当はもっと形式ばった書き方[^bnf]があるんですがこの本ではこれくらいで。

[^bnf]: 代表的なものに [バッカス・ナウア記法](https://ja.wikipedia.org/wiki/%E3%83%90%E3%83%83%E3%82%AB%E3%82%B9%E3%83%BB%E3%83%8A%E3%82%A6%E3%82%A2%E8%A8%98%E6%B3%95)（Backus-Naur Form: BNF 記法）があります。

「式」というのは評価器に渡す AST 全体のことを指しますが、今のところは数ひとつだけを含む単純なものだけを扱います。「プライマリ式」というのは「基本式」「一次式」などとも呼ばれ、式の基本要素を表します。細かい定義はしないで、コードに語ってもらいましょう。

```py
    def parse(self):
        expr = self._expression()
        assert self._current_token() == "$EOF", \
            f"Extra token @ parse(): {self._current_token()}"
        return expr
```

`parse()` は `Parser` クラスで唯一の外部から呼ばれるメソッドです。ここでソースコード全体の構文解析を行って結果を返します。

メソッドの中身は「プログラム」は「式」である、ということを表現しています。`expr = self._expression()` で `_expression()` に「式」の処理を任せます。`_expression()` から帰ってくると、トークン列をすべて消費して現在位置のトークンは `"$EOF"` になっているはず。そうなってなかったらエラーにします。`"$EOF"` になっていれば正常に構文解析ができたということで `expr` を返します。

```py
    def _expression(self): return self._primary()
```

`_expression()` では「式」の構文解析を行います。「式」は「プライマリ式」、なので `_primary()` に「プライマリ式」の処理を任せ、結果をそのまま返します。

```py
    def _primary(self):
        match self._current_token():
            case int(): return self._current_and_advance()
            case invalid:
                assert False, f"Invalid token @ _primary(): {invalid}"
```

`_primary()` ではプライマリ式の構文解析を行います。

「プライマリ式」は「数」、なので、現在のトークンが数（`int`）であるかを調べ、数ならば現在位置をひとつ進め（この場合は `"$EOF"` に進むはず）、数のトークンをそのまま返します。今回の実装では、あとは `_expression()`・`parse()` と戻って呼び出し元に `<数>` という AST が返ります。

やってることはものすごく単純なので、書いてあることは理解できると思いますが、これでなにか解析してるの？とまだちょっとピンとこないところがあるかもしれません。先へ進めば解析してる感が出てきますのでそこは心配無用ですが、ちょっと具体的にイメージしてみましょう。`parse([2, "$EOF"])` と呼び出された場合、

`parse()`: ボスから「プログラム」をもらったぞ。これは「式」なはずだから「式」の処理頼む。
`_expression()`:「式」をもらったぞ。これは「プライマリ式」なはずだから「プライマリ式」の処理頼む。
`_primary()`:「プライマリ式」をもらったぞ。これは「数」なはず・・・確かにそうだ。ということはこの「プライマリ式」の AST は `2` です、ボス！
`_expression()`: おおそうか。「プライマリ式」の AST が `2` なので「式」の AST も `2` です、ボス！
`parse()`: おおそうか。「式」の AST が `2` なら「プログラム」の AST も `2` だな。「プログラム」にはもう余分な項目は残ってないな。よし。この「プログラム」の AST は `2` です、ボス！

という感じでボスから下っ端に指令が伝わって解析が行われます。トップダウンな感じですね？このトップダウンな感じが「再帰下降」の「下降」が意味するところです。「文法をほぼそのままコードにできて」というところの片りんも感じられると思います。今後構文解析を拡張していくときもこのようなイメージを持っていると何をやっているか把握しやすいと思います。あと「再帰」については少し後の節で。

なお今回の範囲ではボス連中は部下へほぼ丸投げしてるだけですが今後はボス連中の仕事も増えていきます。

`case invalid:` について補足です。`invalid` はただの変数で、特殊な意味はありません。`case <変数>:` と書くとこれはなんにでもマッチし、変数には `match` の対象（ここでは `self._current_token()`）の値が入ります。そうすると `assert False, f"Invalid token @ _primary(): {invalid}"` でメッセージの中にエラーになったトークンが埋め込まれます。以下のように書いても同じことになります。

```py
            case _:
                assert False, f"Invalid token @ _primary(): {self._current_token()}"
```

ここでは `_current_token()` の呼び出しを 1 回減らせるな、ここへ来たら Invalid（不正）なんだ、という意図も伝わりやすいかな、くらいの感じでこういう書き方にしています。

構文解析器ができましたので、呼び出し側を作ります。`Parser` オブジェクトを作ってトークン列を与え、`parse()` を呼ぶだけ。

```diff py
 class Interpreter:
     def scan(self, src):
         return Scanner(src).tokenize()
 
+    def parse(self, tokens):
+        return Parser(tokens).parse()
+
     def eval(self, expr):
         return Evaluator().eval(expr, self._env)
```

実行例です。

```py
    print("Parse numbers:")

    print(toil.parse([2, "$EOF"])) # -> 2
    print(toil.parse([23, "$EOF"])) # -> 23

    # print(toil.parse([2, 34, "$EOF"])) # -> Extra token
    # print(toil.parse(['$EOF'])) # -> Invalid token
```

成功する例についてはもう問題ないでしょう。エラーになる例についてはソースコードと見比べつつどこでエラーになるか確認してください。

さて、字句解析と構文解析ができた、ということは！
