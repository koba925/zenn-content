title: "2.1 数を読む（字句解析）"
---

文字列で書かれたソースコードを AST に変換するのが本章のテーマですが、これは 2 段階に分けて行うのが王道です。

第 1 段階では文字列を意味のある字句（トークン：Token）に分けます。これを字句解析と言います。英語では Lexical Analysis や Tokenize と言います。字句解析を行うプログラムを字句解析器と呼んでいきますが Tokenizer や Lexer とも呼ばれます。

Toil の字句解析では、たとえば `"if a == 2 then a else b end"` を入力すると `["if", "a", 2, "then", "a", "else", "b", "end", "$EOF"]` と出力します。最後の `"$EOF"` というのは、トークンの最後を示すマークです。

ここで、出力の中の `2` は文字列の `"2"` ではなく数としての `2` になっていることに注意してください。字句解析ではただ文字列を切るだけではなく、それがどういう種類の字句であるかも判別します。この場合は数の `2` であると判別されたわけです。

`"if"` や `"a"` は文字列ではありますが、字句解析の出力としては単なる文字列ではなく、「これは if 式である」「これは a という変数である」という意味を持った「名前」として扱います。こういった「名前」を「識別子」（Identifier）と呼びます。`"if"` のような文法上特別の意味を持つ「名前」はさらに分けて「キーワード」（Keyword）として扱ったり、あるいは言語側で使うからほかの用途に使っちゃだめだよ！という意味で「予約語」（Reserved word）と呼んだりしますが Toil では分類せずにどれも「識別子」として扱います。

このように、トークンには種別だとかあとどのファイルの何行目の何文字目にあるだとかいう情報を持たせるために本格的な言語では識別子用のクラスだとか `if` 用のクラスだとかを作るのが普通ですが、Toil では数は数、文字列は識別子、という約束でいきます。

字句解析の第一歩は、数字の列を読んで数（`int`）を出力するところから始めましょう。「数字」と「数」を使い分けていることに注意してください。「数字」と言ったら `"1"`・`"2"`・・・といったひとつの文字、「数」といったら `2`、`23` のような（1 けたとは限らない）数を指します。

Toil では「数」は「数字で始まり、そのあとに 0 文字以上の数字が並んだ文字列」で表されるものとします。つまりごく普通の整数のみ扱うということですね。小数点だとかマイナスだとか `10e-5` のような表現は扱いません。

もう一つ大事なことは、数以外に数字で始まるものはない、ということです。たとえば `2or3` といった識別子を許すことにすると `"2"` を読んだ時点ではここから始まる文字列が数字なのか識別子なのか判断することができず、判定が複雑になります。JavaScript では 16 進数を `0x12ab` のように `0x` で始めますが、これも先頭の 0 を読んだ時点で数だと判定できるという条件を満たした表し方です。

これ以外にも列の最初の要素を読んだだけでその列がどういうものかわかるようにしておくことで判断を簡単にするというのは随所に出てきます。Toil はそういうとこだらけです。


`"2"` を `2` に直すだけなら `int()` を呼ぶだけでいいのですが、今後いろいろなものを読むことになるため、一文字ずつ読み進めるループを作ります。

toil.py の先頭に以下のコードを追加してください。

数字を読むだけにしては図体が大きくてちょっとインクリメンタルな感じじゃないのが申し訳ないんですが、ちょこちょこつけたしては解説するよりもここではひとまとめに基本構造を説明したほうがわかりやすいかなと。

```diff py
+class Scanner:
+    def __init__(self, src):
+        self._src = src
+        self._start_pos = 0
+        self._current_pos = 0
+        self._tokens = []
+
+    def tokenize(self):
+        while True:
+            self._start_pos = self._current_pos
+            match self._current_char():
+                case "$EOF":
+                    self._tokens.append("$EOF")
+                    break
+                case c if c.isdecimal(): self._number()
+                case invalid:
+                    assert False, f"Invalid character @ tokenize(): {invalid}"
+
+        return self._tokens
+
+    def _number(self):
+        while self._current_char().isdecimal(): self._advance()
+        self._tokens.append(int(self._lexeme()))
+
+    def _lexeme(self):
+        return self._src[self._start_pos:self._current_pos]
+
+    def _advance(self): self._current_pos += 1
+
+    def _current_char(self):
+        if self._current_pos < len(self._src):
+            return self._src[self._current_pos]
+        else:
+            return "$EOF"
```

パーツに分けて説明していきます。ちょっと説明が前後するところがあるので、わからないところがあってもいったん `Scanner` クラスの説明の最後までざっと読んでみてください。そのあと戻って読み直したほうがわかりやすいと思います。

```py
class Scanner:
    def __init__(self, src):
        self._src = src
        self._start_pos = 0
        self._current_pos = 0
        self._tokens = []
```

`Scanner` オブジェクトを初期化します。`_src` はソースコードの文字列、`_start_pos` はトークンの開始位置、`_current_pos` は今読んでいる位置、`_tokens` はトークンの列で、字句解析器の出力となるものです。

次は下のほうに移ってユーティリティメソッドの説明です。

```py
    def _advance(self): self._current_pos += 1

    def _current_char(self):
        if self._current_pos < len(self._src):
            return self._src[self._current_pos]
        else:
            return "$EOF"
```

`_advance()` は読んでいる位置をひとつ進めます。

`_current_char()` は現在位置にある文字を返しますが、現在位置が文字列の最後を過ぎていたらそのしるしとして `"$EOF"` を返します。

```py
    def _lexeme(self):
        return self._src[self._start_pos:self._current_pos]
```

レクシム（Lexeme）はトークンに近い意味ですが、ここではレクシムはソースコード上に現れる文字列を、トークンはレクシムを解釈してこれは数だとかこれは識別子だと判断したもの、と使い分けています。

`_lexeme()` ではソースコードの `self._start_pos` 文字目から `self._current_pos` の手前までの文字列を返します。

```py
    def tokenize(self):
        while True:
            self._start_pos = self._current_pos
```

`tokenize()` は `Scanner` クラスで唯一ほかから呼ばれるメソッドです。

`while True:` から字句解析のメインループが始まります。`self._start_pos = self._current_pos` では、あとで `_lexeme()` を使って文字列を切り出せるよう、ここからトークンが始まるよ、という位置を記録しています。

```py
            match self._current_char():
```

`_current_char()` に応じた処理を行います。

```py
                case "$EOF":
                    self._tokens.append("$EOF")
                    break
```

`_current_char()` が `"$EOF"` であればソースコードの最後に達したということなので `_tokens` に `"$EOF"` を追加してループを抜けます。

```py
                case c if c.isdecimal(): self._number()
```

ガード構文ですね。Toil の文法では読んだ文字が数字であればそれは数のトークンだと判断できますので `_number()` で処理します。

```py
                case invalid:
                    assert False, f"Invalid character @ tokenize(): {invalid}"
```

どれにも当てはまらなければ不正な文字を読んだということでエラー。

```py
        return self._tokens
```

ループを抜けたら字句解析は終わりです。`_tokens` を返します[^on-demand-tokenize]。

[^on-demand-tokenize]: Toil の処理系では、ソースコード全体を字句解析した後すべてのトークンを配列に入れて返していますが、構文解析器（次節で解説）の要求に応じて１トークンずつ返す流儀もあります。昔はメモリ容量が限られていてソースコードやトークンを全体をすべてメモリに置いておくのはもったいないという事情がありました。今はメモリは潤沢ですし、シンプルさや字句解析結果の見やすさを優先してこのようにしています。

```py
    def _number(self):
        while self._current_char().isdecimal(): self._advance()
        self._tokens.append(int(self._lexeme()))
```

`while` では、現在の文字が数字である限り `_advance()` を読んで位置を進めます。数字以外の文字（`"$EOF"` も含む）が現れたら `while` を抜け、`_tokens` に数のトークンを追加します。`self._lexeme()` で今読んでいる文字列を切り出し、 `int()` で文字列を数（`int`）にすることで数のトークンにしています。

これで `Scanner` クラスの説明は終わりです。先頭の文字を見てそれが何のトークンであるか判断し、トークンの最後まで読み進めてソースコードから切り出し `_tokens` に追加する、というのが基本パターンで今後も出てきますのでしっかりマスターしておいてください。

`Interpreter` クラスに字句解析を行うメソッドを追加すれば今回の変更は終わりです。

```diff py
 class Interpreter:
     ...

     def _builtins(self):
         ...

+    def scan(self, src):
+        return Scanner(src).tokenize()
```

これで動くはず。

```py
    print("Scan numbers:")

    print(toil.scan(r""""""))  # -> ['$EOF']
    print(toil.scan(r"""2"""))  # -> [2, '$EOF']
    print(toil.scan(r"""02"""))  # -> [2, '$EOF']
    print(toil.scan(r"""23"""))  # -> [23, '$EOF']
```

今後、Toil のソースコードは主に raw トリプルクォート文字列  `r"""..."""` のような文字列で書きます。ソースコード内にはクォーテーション（`'` や `"`）やバックスラッシュ（`\`）を書いたり改行を入れたりすることがありますが、raw トリプルクォート文字列であればほとんどのケースでそのまま、エスケープしないで書けるためです。たまにはあえてエスケープしたいこともあり、raw でないトリプルクォート文字列  `"""..."""` を使うこともあります。必要な時だけにしないのは、忘れるの防止と、見た目揃えたいくらいの理由でどうしてもそうでないといけないというわけではありません。

プログラムのほうで解説しているので結果は納得できるのではないかと思います。しいていえばゼロで始まる数も特に特別扱いしないで `"02"` が `2` になっていることくらいでしょうか。ここも半分は Python の `int()` の仕様に依存しているところです。


```py

    # print(toil.scan(r"""2$"""))  # -> Invalid character
```

不正な文字が入力されたらエラーにします。

## コラム：Python の文字列

`_start_pos` にトークンの開始位置を覚えておき、`_current_pos` を進めて、トークンの終わりが来た時に `self._src[self._start_pos:self._current_pos]` でレクシムを切り出す、というやり方にちょっと疑問を持った方もいるかもしれません。

開始位置を覚えておかなくても、`self._token = ""` から始めて 1 文字進める、つまり `_advance()` を呼ぶたびに `self._token += _current_char()` すれば素直にトークンの文字列ができていくのでは、と。

そんなふうに考えていた時期が俺にもありました。

位置を覚えておくっていうのはあんまりやりたくないことで、`for (i = 0; i < n; i++) { ... arr[i] ... }` と書くより `for e in arr { ... e ... }` って書くほうがいいというのと同じ感覚です。

ですが、Python の文字列で `self._token += _current_char()` のように 1 文字ずつ足していくというのはよくない書き方とされているのです。

なぜかというと、Python の文字列はイミュータブルだからです。イミュータブルというのは、一度作ったらその後値を変えることができないということです。配列では `a = [2, 3, 4]; a[1] = 5` とできますが文字列で `s = "abc"; s[1] = "d"` とはできませんね。

そうすると何が起こるかというと、たとえば `"abcde"` という文字列に `"f"` を足そうとしたら `"abcde"` の末尾に `"f"` をくっつけるのではなく、もとからある `"abcde"` はそのままにして新たに `"abcdef"` という文字列を作るのです。

`"a"` から順番に 1 文字ずつ足して `"abcdef"` を作ろうとしたらこんなことになってしまいます。

`"a"` ← 残ったまま
`"ab"` ← 残ったまま
`"abc"` ← 残ったまま
`"abcd"` ← 残ったまま
`"abcde"` ← 残ったまま
`"abcdef"`

メモリ的にも性能的にもよろしくありません。メモリはいずれガベージコレクションでお掃除されますが。

それでも、メモリ何ギガというこのご時世で数文字程度は誤差にもならないレベルだし、わかりやすさを優先して `self._token += _current_char()` で書こうか・・・とも思ったんですがよくない書き方を伝えるのもよくないし、ということで定石にのっとった書き方にしています。

## コラム：番兵

トークン列の最後に `"$EOF"` という文字列を追加しています。これは「番兵」（Sentinel）と呼ばれる技法です。`_current_char()` がソースコードの末尾で `"$EOF"` を返すのも番兵の変形といってよいでしょう。

番兵を使わなくても位置と長さの比較で最後まで到達したか判定できますが、ループの終了やら要素へのアクセスやらを行うたびに位置を確認するのは煩雑です。番兵を使うとすっきり書けることが多く、また比較して判定するのを減らすことにもなるので性能にも好影響が期待できます。

注意点として、番兵に使う値は、普通に現れる値と絶対にカブらないようにしないといけません。`_current_char()` では通常の文字は 1 文字ずつしかとらないので、`"$EOF"` には絶対にマッチせず安全です。
