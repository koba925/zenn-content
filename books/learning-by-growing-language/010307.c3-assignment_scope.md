次は代入とスコープです。代入は `define()` の代わりに `assign()` を呼ぶだけなのでなんてことはありません。スコープをどうするかが本節のテーマです。

新たなスコープを作るときに `Environment(self._env)` とすることもこれまでどおりで、`self._env = Environment(self._env)` として `_env` を書き換えてやれば次からはそのスコープの中で命令が実行されます。

じゃああとは何かというと入ったスコープから出たときにどうやって復帰するかです。さっきまで使っていた `_env` は上書きしてしまったので消えています。復帰するには使っていた `_env` を覚えておく仕組みが必要になります。

そのために、仮想マシンにもうひとつ実行制御用のスタック `_ctrl_stack` を用意してそこに過去の `_env` を記録しておくようにします。

```diff py
 class VM:
     def __init__(self, code, env):
         self._code = code
         self._env = env
         self._ip = 0
         self._stack = []
+        self._ctrl_stack = []
```

`_ctrl_stack` をスタックにしているのは、スコープの中でまたスコープを作るといったように入れ子になる可能性があるからです。入れ子を見たらスタックを思い出しましょう。

作戦が決まれば実装は難しくありません。

```diff py 
     def execute(self):
         while (inst := self._code[self._ip]) != ("halt",):
             self._ip += 1
             match inst:
                 ...
                 case ("pop",): self._stack.pop()
+                case ("enter_scope",):
+                    self._ctrl_stack.append(("scope", self._env))
+                    self._env = Environment(self._env)
+                case ("leave_scope",):
+                    _, self._env = self._ctrl_stack.pop()
                 case ("def", name): self._env.define(name, self._stack[-1])
+                case ("set", name): self._env.assign(name, self._stack[-1])
                 case ("get", name): self._stack.append(self._env.val(name))
                 ...
             ...
+        assert len(self._ctrl_stack) == 0, \
+            f"Invalid control stack state @ execute(): {self._ctrl_stack}"
         assert len(self._stack) == 1, \
             f"Invalid stack state @ execute(): {self._stack}"
         return self._stack.pop()
```

命令が `enter_scope` であれば現在の `_env` を `_ctrl_stack` に積んでから `_env` を新しい環境で上書きします。`leave_scope` なら `_ctrl_stack` から一つ前の環境を取ってきて `_env` に設定します。

`set` はいいですね[^dont]。

[^dont]: 気になることがあるとすれば `define` が `def` なら `assign` は・・・？となるところですがそれはお下品なのでダメです。

プログラム終了時にはすべてのスコープから出ているはずなので `len(self._ctrl_stack) == 0` かどうかを確認しています。

コンパイラ側も `assign` の動作は問題ないでしょう。

```diff py
class Compiler:
     ...
 
     def _expression(self, expr):
         match expr:
             ...
             case ("define", [name, expr]):
                 self._expression(expr)
                 self._emit("def", name)
+            case ("assign", [name, expr]):
+                self._expression(expr)
+                self._emit("set", name)
+            case ("scope", [body_expr]): self._scope(body_expr)
             case ("seq", exprs): self._seq(exprs)
             ...

+    def _scope(self, body_expr):
+        self._emit("enter_scope")
+        self._expression(body_expr)
+        self._emit("leave_scope")

     def _seq(self, exprs):
         ...
```

`scope` も、本体の式を実行する前に `enter_scope` であらたなスコープを作ってそこに入り、実行後に `leave_scope` で元のスコープに戻るよう中間コードを出力しているだけです。プログラム終了時には `len(self._ctrl_stack) == 0` となっているはずだということがこのコードからわかります。

実行例です。

```py
    print("Assignment and Scope:")

    print(toil.ast(r""" a = 3 + 4 """))
    # -> ('assign', ['a', ('add', [3, 4])])
    print_code(toil.code(r""" a = 3 + 4 """))
    # ->   0: ('const', 3)
    # ->   1: ('const', 4)
    # ->   2: ('add',)
    # ->   3: ('set', 'a')
    # ->   4: ('halt',)

    print(toil.run(r""" a """)) # -> 7

    print(toil.run(r""" a """)) # -> 4

    print(toil.ast(r""" scope a + 3 end """))
    # -> ('scope', [('add', ['a', 3])])
    print_code(toil.code(r""" scope a + 3 end """))
    # ->   0: ('enter_scope',)
    # ->   1: ('get', 'a')
    # ->   2: ('const', 3)
    # ->   3: ('add',)
    # ->   4: ('leave_scope',)
    # ->   5: ('halt',)

    print(toil.run(r""" a := 2; scope scope a + 3 end end """)) # -> 5

    print(toil.run(r""" a := 2; scope a := 3 end """)) # -> 3
```

## コラム：環境の持ち方

> 復帰するには使っていた `_env` を覚えておく仕組みが必要になります。

でも第 1 章ではそんなこと気にしてなかったけど？と思った人もいるでしょう。

`Evaluator` の `scope` は以下のように実装されていました。どこにも環境を元に戻すコードはありません。

```py
    def eval(self, expr, env):
        match expr:
            ...
            case ("scope", [body_expr]):
                return self.eval(body_expr, Environment(env))
            ...
```

関数呼び出しの処理でもスコープを作る処理がありますがやはり環境を戻す処理はありません。

```py
    def _op(self, op_expr, args_expr, env):
        ...
        match op_val:
            ...
            case ("closure", [params, body_expr, closure_env]):
                new_env = Environment(closure_env)
                new_env.bind(params, args_val)
                return self.eval(body_expr, new_env)
            ...
```

これは、これらの関数において `env` が引数であり、引数は関数呼び出しごとに Python がコールスタック上にローカルな変数として保存してくれていて、そもそも上書きされないからです。変わらないから戻す必要もない。

これが `eval()` でわざわざ `env` を引数で持ち回っている理由です。`Evaluator` クラスでも、環境を `self._env` のような属性に覚えておき、引数に入れて持ち回らなくても参照できるようにすることはできます。ただしその場合は再帰的に呼び出される関数において正しいタイミングで `_env` を元に戻してやる必要があり、戻し忘れるとバグになります。引数で持ち回っていればそういうリスクを避けることができるというわけです。

第 3 章における実装ではそういうリスクを冒している面はありますが、一方では引数に入れて Python まかせだったスコープの管理を、明にスタックを用いて自分で環境を戻してやることで、一段階深くスコープのしくみを理解できたとも言えます。また、実際のコンパイラのスコープや関数呼び出しのしくみにも通じるものがあります。
