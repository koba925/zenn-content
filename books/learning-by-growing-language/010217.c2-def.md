title: "2.17 def-do-end"
---

前節で `double` という関数を `double := func x do x * 2 end` のように書けるようにしました。でもなんかね、とすっきりしない感を感じてる人もいるかと思います。

`def double(x) do x * 2 end` と書けるようにしましょう。これなら普通っぽいですよね。

これは、`def double(x) do x * 2 end` というソースコードを読んだら `double := func x do x * 2 end` と同じ `("define", ["double", ("func", [["x"], ("mul", ["x", 2])])])` という AST を出力するようにしてやるだけで、`eval()` には手を加えずに機能を追加できます。やることは同じでソースコード上の見かけが違うだけです。こういうのを構文糖衣（Syntax Sugar）と言います。`def double(x) do x * 2 end` は `double := func x do x * 2 end` の構文糖衣だ、というわけです。

ただ、これはそういう実装方法もある、ということで必ずそうしなければならないというものではありません。Python の `def` はただの構文糖衣ではありません。Python の `def` で定義した関数は自分の名前を知ってます（だからコールスタックに関数の名前が表示できる）。

実装ですが、`def double(x) do x * 2 end` の `double(x)` の部分が関数呼び出しの形になっていることがミソです。`def` を読んだ後、`_expression()` を呼んでやると `double(x)` を関数呼び出しの `<式>` として読んだ `("double", ["x"])` が返ってきます。これ、ちょうど `("define", ["double", ("func", [["x"], ("mul", ["x", 2])])])` で使いたい `"double"` と `["x"]` が含まれていますね。専用の読み込みメソッドを作らなくて済みます。

そのことを利用するとこのようなコードになります。

```diff py
 class Parser:
     ...
 
     def _primary(self):
         match self._current_token():
             ...
             case "func": return self._func()
+            case "def": return self._def()
             case "scope": return self._scope()
             ...

     ...

     def _func(self):
         ...
 
+    def _def(self):
+        self._current_and_advance()
+        call_expr = self._expression()
+        self._consume("do")
+        body_expr = self._expression()
+        self._consume("end")
+        match call_expr:
+            case (name, params):
+                return ("define", [name, ("func", [params, body_expr])])
+            case str():
+                return ("define", [call_expr, ("func", [[], body_expr])])
+            case _:
+                assert False, f"Invalid def syntax @ _def(): {call_expr}"

     def _scope(self):
         ...
```

`double(x)` の部分を `_expression()` で読んだらとりあえず `call_expr` に覚えておき、`end` まで読んでから `call_expr` の形によって処理を分けます。

関数の形だったら `case (name, params)` で `name` に関数名 、`params` に引数リストを入れて `("define", [name, ("func", [params, body_expr])])` を返します。

次の `case str()` は `call_expr` に単なる文字列が入っているとき、つまり `def foo do 2 end` のようなコードにマッチします。これは引数なしの関数を定義しようとしているとみなし、`("define", [call_expr, ("func", [[], body_expr])])` を返します[^readability-vs-complexity]。`def foo() do 2 end` のように書いても問題なく、この場合はひとつめの `case` で処理されます。

[^readability-vs-complexity]: こんな風に、ソースコードをちょっとみやすくするために例外的なルールをいれることは往々にしてプログラムが思わぬところで思った以上に複雑になったり最悪文法が破綻したりするものですが Toil はどうやらうまくいってるようです。

`_expression()` で読んでいるので `call_expr` にはどんな形の式でも入ってくる可能性がありますが、上のふたつの `case` 以外はエラーにしています。

実行例です。

```py
    print("Function Definition:")

    print(toil.ast(r""" def f do 2 end """))
    # -> ('define', ['f', ('func', [[], 2])])
    toil.walk(r""" def f do 2 end """)
    print(toil.walk(r""" f() """)) # -> 2

    print(toil.ast(r""" def f() do 3 end """))
    # -> ('define', ['f', ('func', [[], 3])])
    toil.walk(r""" def f() do 3 end """)
    print(toil.walk(r""" f() """)) # -> 3

    print(toil.ast(r""" def f(a) do a + 2 end """))
    # -> ('define', ['f', ('func', [['a'], ('add', ['a', 2])])])
    toil.walk(r""" def f(a) do a + 2 end """)
    print(toil.walk(r""" f(3) """)) # -> 5

    print(toil.ast(r""" def f(a, b) do a + b end """))
    # -> ('define', ['f', ('func', [['a', 'b'], ('add', ['a', 'b'])])])
    toil.walk(r""" def f(a, b) do a + b end """)
    print(toil.walk(r""" f(2, 3) """)) # -> 5

    print(toil.walk(r"""
        a := 2;
        def f do a end;
        def g do a := 3; f() end;
        g()
    """)) # -> 2

    # toil.walk(r""" def do a end """) # -> Expected do
    # toil.walk(r""" def f(a) a end """) # -> Expected do
    # toil.walk(r""" def f(a) do end """) # -> Expected end
    # toil.walk(r""" def f(a) do a """) # -> Expected end
    # toil.walk(r""" def 2 do a end """) # -> Invalid def syntax
```

引数が 0 個（カッコなし）、0 個（カッコあり）、1 個、2 個のときにちゃんと処理できてますね！

これでかなり普通っぽいソースコードが書けるようになってるはずです。次はいろいろ書いてみましょう。
